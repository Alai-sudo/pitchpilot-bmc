PitchPilot 功能说明 PM / BD / 设计 / 新同事的功能全量参考

一句话　对话式 AI Pitch 生成器，从 Confluence / 文件 / 对话出发，自动产出 HTML 或 PPTX deck
核心承诺　从 0 到一份可用 Pitch 材料 ≤ 15 分钟 更新　2026-05-19 双部署　EKS+Arx（多 agent 完整版）/ Cloudflare Workers（轻量 SaaS 版）

01工作流：三阶段

每个 deck 都按 gather → plan → preview 三阶段推进。阶段不是强制的——用户可随时跳转——但模型按当前阶段调整发言策略，UI 也会切换主面板。

1

Gather · 收集

弄清楚要做什么。受众、目标、场景、关键信息。

  • 对话收集 audience / goal / scene / key messages
  • 用户可粘贴图片 / 拖拽文件 / 引用 Confluence 页面
  • 右侧主面板：ConceptCloud 实时可视化已收集到的概念
  • 触发 talent：pp-gather + pp-confluence + pp-url-parser
  • "准备好了"由 pp-planner 判断并主动进入 plan 阶段
2

Plan · 策划

把素材组织成有叙事逻辑的 storyboard。

  • 右侧主面板切换为 storyboard 编辑器（Tiptap / Vditor）
  • AI 生成 markdown 草稿，用户可编辑 / 让 AI 重写局部
  • 触发 talent：pp-planner + pp-narrative + pp-speaker
  • 编辑时存在 AI editing lock，避免人和 AI 同时改
  • storyboard 是 deck 的一等公民，会被持久化到 deck 的 storage
3

Preview · 构建

把 storyboard 转为可视化幻灯片。

  • 进入时弹出 BuildModePicker，选 HTML 或 PPTX
  • 右侧切换为 PreviewPanel，单页渐进式渲染，带 building overlay
  • HTML 模式：触发 pp-deck-builder + pp-deck-designer
  • PPTX 模式：触发 pp-deck-builder-pptx，按 9 种 layout 模板装配
  • 逐张幻灯片可被点击进入 inline 编辑或让 AI 修改

02创作模式：两个维度

每个 deck 有两个独立维度的"模式"——多 agent 协同方式（chat_mode）+ 输出形态（build_mode）。两者正交，可任意组合。

Chat Mode多 agent 协同

默认Group · 群聊
所有在线 tulpa 都收到每条消息，按各自人设决定是否回应（通过 [SILENT] 协议保持安静）。消息会被注入"成员名册 + 群规"上下文。适合多 agent 自主分工。
Relay · 中继
按当前 phase 路由到单个 agent：gather / plan → planner；preview → builder。适合结构化、可预测的流水线。

Build Mode输出形态

默认HTML
Web-native 幻灯片，单 fragment 文件，可在浏览器直接演讲。支持复杂动画、自定义字体、SVG 图表。分享一个链接即可。
PPTX
原生 .pptx 文件，9 个 layout 模板装配。可下载、可在 PowerPoint 离线编辑。客户要"可编辑文件"时用这个。生成时需选 theme。

03内置 Talents：12 个

Talent 是 PP 的核心扩展点——每个 talent 是 AI agent 的一项能力（skill / 工具链 / 知识包）。Agent 会按对话上下文自动激活相关 talent。

Talent 类型 职责 典型触发场景
pp-gather skill 从对话提取 audience / goal / key messages，构建 ConceptCloud 用户首次描述 pitch 需求时
pp-narrative skill 叙事框架库：story 弧线、受众驱动叙事 plan 阶段，组织 storyboard 时
pp-sales-pitch skill 销售 Pitch 专家：deal-stage 框架、竞争定位 用户明确"这是销售用 deck"时
pp-speaker skill 生成演讲备注、stage direction storyboard 完成后或用户索要 speaker notes
pp-deck-builder skill HTML 幻灯片工程：fragment 格式、manifest.json、生命周期 preview 阶段 + build_mode = html
pp-deck-builder-pptx skill 原生 PPTX 工程：按 9 个 layout 输出 slide-NN.json preview 阶段 + build_mode = pptx
pp-deck-designer skill Dify Swiss 设计系统：白底、三色调色板、Söhne+MiSans、非对称排版、零装饰 所有 build 阶段的视觉约束
pp-confluence skill 搜索和读取 Confluence 页面 用户提到 wiki / Confluence / 公司知识库时
pp-url-parser skill 检测消息里的 URL，抓取并结构化提取 消息含 http(s):// 链接时
pp-web-research skill Web 搜索 + 综合：市场数据、竞品、引用 需要外部素材但 Confluence 没有时
pp-cli toolchain pp CLI：save / sync / download / ls，deck 文件锁、Confluence 代理 所有 agent 内部 IO 操作
pp-dify-context knowledge Dify 公司身份、使命、愿景、价值观、tone 需要"on-brand"输出的所有场景（默认加载）

04输入与上下文

PP 不让用户从空白开始——所有创作都建立在真实素材上。四种输入方式覆盖企业知识库、本地文件、外部链接、Web 现场搜索。

Confluence 集成

  • OAuth 2.0 接入企业 Atlassian 账号
  • 侧滑 ConfluencePanel：浏览 space / 搜索 / 选择页面
  • 页面以引用方式绑定到 deck，自动转 markdown 注入对话
  • 支持 @mention 解析、附件 / image 拉取
  • 页面更新后可重新拉取（不自动轮询，按需）

文件上传

  • PDF / Word / Excel / 图片 / Markdown / TXT
  • 拖拽到聊天框，或从剪贴板粘贴图片
  • 上传后立即作为 agent context；持久化到 deck storage
  • 文件管理：FileViewerDrawer 查看 / 删除 / 重新发送

URL 解析

  • pp-url-parser 自动检测消息里的 URL
  • 抓取页面 → 提取标题、正文、关键数据
  • 适合引用产品文档、竞品页、新闻稿

Web 搜索

  • pp-web-research 在 Confluence 没有素材时兜底
  • 搜索 + 综合多源 → 结构化输出（带引用）
  • 适合市场数据、竞品分析、行业数字

05协作与多租户

PP 是多用户系统。组织粒度从高到低：Workspace（团队）→ Folder（项目分组）→ Deck（单个 pitch）→ Deck Members（per-deck 权限）。

单位 说明 权限模型
Workspace 团队容器。一个用户可属于多个 workspace，每个 workspace 独立计费、独立 API key、独立成员名单 owner / admin / member
Folder Workspace 内部的 deck 分组，用于按项目 / 客户 / 季度组织 继承 workspace
Deck 单个 pitch 文档。属于 workspace + 可选 folder workspace 成员默认可见
Deck Members per-deck 显式成员（细粒度覆盖 workspace 默认） owner / editor / viewer
Share Link 对外公开访问链接，无需登录 token-based，可随时撤销

06编辑、预览与版本

deck 是活的文档——AI 写、人编辑、版本追溯。三个核心交互面板加一套版本系统。

ConceptCloud

  • gather 阶段的主面板
  • 渐变文字 + 发光节点的概念图
  • 实时反映 agent 已"听到"什么
  • 用户点节点 → 注入对话焦点

Storyboard Editor

  • plan 阶段的主面板
  • 双引擎：Tiptap（富文本）+ Vditor（markdown）
  • 预览 / 编辑切换
  • AI editing lock：人 / AI 互不踩

PreviewPanel

  • preview 阶段的主面板
  • 单页渐进渲染 + breathing overlay
  • HTML / PPTX tab 切换
  • SlideNav：缩略图导航

版本历史

每个 deck 维护版本快照。GET /:id/versions 列表、POST /:id/versions 创建快照、PUT /:id/versions/current 切回旧版本。 版本里包含 storyboard、artifacts、context。适合"客户改方向了，回到三天前那版"场景。

07输出

三种输出形态可同时存在：浏览器渲染、可下载 PPTX、speaker notes 文本。

HTML Deck

  • 单 fragment HTML，包含所有 CSS / JS / 字体
  • 浏览器直接演讲，按 ←/→ 切换
  • 支持自定义动画、SVG 图表、互动元素
  • 分享链接对外公开访问（无需登录）
  • 导出 GET /:deckId/export

PPTX Deck

  • 原生 .pptx 文件，可在 PowerPoint / Keynote 编辑
  • 主题 theme 在 BuildModePicker 选择
  • 下载 GET /:deckId/export.pptx
  • 9 个 layout 模板（见下）
cover bullets cards big_numbers comparison quote image_caption timeline closing

Speaker Notes

pp-speaker talent 自动为每张幻灯片生成演讲备注 + stage direction（"这里停顿，让数字落地"）。 内嵌在 PPTX notes 区，HTML 模式下可单独查看。

08管理与运维

Admin Dashboard 是工作区拥有者的控制面板——配置 LLM 凭证、agent、模板、用量。

Workspace 管理员能做什么

  • 配置 Anthropic / OpenAI API key（覆盖系统默认）
  • 切换默认模型（Opus / Sonnet / Haiku）
  • 邀请 / 删除成员，分配角色
  • 查看 Arx agent 实例列表 + 状态
  • 发布自定义 deck 模板到 workspace 内部库
  • 测试 Arx 连通性

用户个人设置

  • 修改个人 profile / 头像
  • 切换默认 workspace
  • 主题模式（亮 / 暗 / 跟随系统）
  • UI 字体大小（小 / 中 / 大）
  • Confluence OAuth 连接管理（多组织）

09两种部署形态

PP 有两套实现，覆盖不同企业需求。Code 共享同一 React 前端，后端 / agent 运行时不同。

完整版EKS + Arx

  • 后端：Node.js + Express + SQLite/Postgres
  • Agent：Arx 调度 Oasis Docker 容器，跑 Claude Code CLI
  • 能力：多 tulpa 协作、agent-in-sandbox、长会话持久化、自定义 talent
  • 适合：内部部署、企业 self-host、需要 agent 工具调用
  • URL：eks.pp.greeep.com（内部 dev）

轻量版Cloudflare Workers

  • 后端：Cloudflare Workers + Durable Objects + D1 + R2
  • Agent：Anthropic API 直连，可选 Sandbox container（v1.1）
  • 能力：每 workspace 自带 API key，AES-GCM 加密存储
  • 适合：SaaS 形态、需要全球边缘部署、单 agent 场景
  • URL：pitchpilot-api.langgenius-opc.workers.dev

10边界与扩展点

不在范围内

  • 不替代深度战略提案（strategic proposal）的手工撰写
  • 不处理最高保密级客户信息（无审计 / 无合规认证）
  • 不是通用 LLM 聊天界面（只服务 pitch 创作场景）
  • 不做实时多人协同编辑（同一 deck 同一时刻一个 active 编辑者）
  • 不内嵌视频 / 复杂 3D（HTML 模式可手插入，PPTX 不支持）

扩展点

  • 自定义 Talent：在 app/templates/ 加 talent.toml + SKILL.md 即可注入新能力
  • Brand Skill 上传：5 分钟把 PDF brand book 转 talent（规划中）
  • Workflow 节点：通过 Arx API 暴露给 Dify workflow 调用
  • PPTX 模板：在 services/pptx-templates/ 加 layout 文件
  • 第三方知识源：在 routes 加新的 connector（Notion / Drive / SharePoint）
